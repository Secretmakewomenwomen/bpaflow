# AI 服务项目面试可背诵稿（30 分钟版）

## 1. 开场 60 秒（先背这段）

这个项目是一个嵌入在业务工作台里的 AI 助手后端，不是独立聊天应用。  
我负责把它从“单轮检索接口”升级成“可多轮、可工具调用、可中断恢复、可追踪”的 Agent 化服务。

技术上我们用的是：

- `FastAPI` 作为接口层
- `LangGraph` 做状态编排和 checkpoint
- `OpenAI tool-calling` 做模型决策
- `PostgreSQL + pgvector + BM25 + 规则召回` 做 RAG
- `SQLAlchemy` 做消息和 trace 持久化
- `SSE` 做流式输出

一句话总结价值：我把这套 AI 服务做成了“模型负责决策，后端负责执行和治理”的可演进架构，既能快迭代，又能在生产环境可控可排障。

---

## 2. 项目目标与业务场景（2-3 分钟）

### 2.1 业务目标

- 用户在工作台内直接提问，不切换系统。
- 普通问题直接聊天回答。
- 涉及资料检索时，返回可追溯的证据片段和相关文件。
- 支持“根据某文件生成流程图”这类需要用户确认的交互。

### 2.2 典型场景

1. 用户问：“最近上传的保险文档有哪些？”  
Agent 调 `list_recent_files`，返回文件列表。
2. 用户问：“XX 接口报错码 E123 在哪份文档？”  
Agent 调 `search_knowledge_base`，走多路召回并给证据片段。
3. 用户问：“根据保险产品设计手册生成流程图。”  
Agent 进入 `waiting_input`，让用户先选文件，再恢复执行生成流程图 JSON。

## 3. 架构总览（3-4 分钟）

### 3.1 分层

1. API 接入层  
路由入口在 `backend/app/api/routes/ai.py`。负责鉴权、接收查询、启动流式响应、恢复中断动作。
2. 编排层  
`LangGraphAssistantService` 把流程固定为四节点：  
`load_history -> run_agent_loop -> build_response -> persist_message`。
3. Agent 运行层  
`AgentFacade + LangGraphAgentRuntime` 负责 ReAct/tool-calling 循环、工具调用、终止控制、trace 记录。
4. 能力层  
工具系统（用户/文件/检索）+ RAG 检索系统（向量 + BM25 + 规则融合）。
5. 持久化层  
`ai_conversation / ai_message / ai_message_reference / ai_agent_trace`。

### 3.2 请求闭环

1. 前端发起 `messages/stream`。
2. 先落用户消息。
3. Agent 执行循环并通过 SSE 回传 `assistant_start / assistant_reasoning / assistant_delta / assistant_done`。
4. 最终 AI 消息落库，引用单独入库，前端拿到完整持久化消息。

## 4. Agent 设计模式（核心面试点，6-8 分钟）

这里建议你直接说“我们不是单一模式，而是组合了 8 种模式”。

### 4.1 ReAct 模式（Thought-Action-Observation）

- Runtime 每轮先让模型产出决策。
- 若是 `tool_call`，执行工具并把 observation 回灌到消息上下文。
- 若是 `final_answer`，直接结束并出最终回复。

价值：模型可以在一轮里动态决定“继续查还是直接答”。

### 4.2 Tool-Augmented LLM（模型决策，后端执行）

- 工具通过统一 registry 注册 schema。
- 模型只负责选工具和传参数。
- 后端 dispatcher 负责参数校验、权限注入、异常归一化。

价值：把“决策智能”和“执行可靠”解耦。

### 4.3 Graph-Orchestrated Agent（图编排）

- LangGraph 固定主图，避免业务扩展导致图爆炸。
- 复杂逻辑收敛在 `run_agent_loop` 单节点内部循环。

价值：图结构稳定，能力迭代快。

### 4.4 Human-in-the-loop Interrupt（人机协同中断）

- “生成流程图”场景先返回 `pending_action`，状态为 `waiting_input`。
- 用户确认 `uploadId` 后走 `resume`，继续同一线程执行。

价值：把必须由人做的选择从模型里剥离，降低误操作风险。

### 4.5 Guardrails（调用前护栏）

- 校验用户上下文是否存在。
- 校验工具名是否匹配。
- 校验必填参数是否完整。
- 可扩展 `requires_confirmation`。

价值：防止模型乱调工具、越权调用。

### 4.6 Termination Controller（终止策略）

- `max_steps`：最大步数。
- `repeated_action_limit`：重复动作阈值。
- `consecutive_empty_limit`：连续空步阈值。

价值：防止 Agent 进入死循环。

### 4.7 Self-correction（有限自纠）

- 参数错误时允许一次 retryable 自纠。
- 第二次仍失败直接终止并返回错误。

价值：提升成功率，同时控制成本和时延。

### 4.8 Observability-first Tracing（可观测优先）

- 记录四个 phase：`reason / action / observation / termination`。
- 每步写 `ai_agent_trace`，不影响主事务（嵌套事务兜底）。

价值：线上排障可回放，可归因。

## 5. RAG 方案讲解（5-6 分钟）

### 5.1 为什么不是单路向量检索

企业场景里很多查询是“编号、错误码、文件名、接口名”，纯语义召回不稳，所以我们做三路融合：

1. 向量召回：语义相关性
2. BM25 召回：关键词匹配
3. 规则召回：文件名/标识符等强字面信号

### 5.2 核心流程

1. Query 分析：提取关键词、标识符、是否要最近、是否指定文件类型。
2. 可选 Query Rewrite：最多改写为 1-4 个检索短语。
3. 三路并行召回并放大候选。
4. chunk 级去重合并（按 `file_id + chunk_index`）。
5. 加权融合 + bonus（最近上传、类型匹配、文件名精确命中）。
6. 阈值裁剪后生成 `snippets + related_files`。

### 5.3 你可以这样回答“效果提升点”

- 多主题查询覆盖率提高（不会被单一路径偏置）。
- 编号类问题命中率提高（规则召回补位）。
- “最近/图片/PDF”这类意图响应更准（bonus 纠偏）。

## 6. 技术难点与我做的决策（3-4 分钟）

### 6.1 难点一：如何平衡“智能性”和“可控性”

决策：模型只管“选什么工具”，执行与校验完全在后端治理。  
结果：可扩展，同时满足生产稳定性。

### 6.2 难点二：流式展示和最终一致性

决策：SSE 按 delta 实时推送，但必须在落库后发 `assistant_done`。  
结果：前端看到的最终内容与数据库一致。

### 6.3 难点三：涉及用户确认的流程

决策：用 LangGraph interrupt/resume，而不是让模型模拟“已确认”。  
结果：关键动作可审计，可恢复，用户体验连续。

## 8. 面试高频 QA（场景 + 技术 + 回答提示）

### Q1：你们这个是不是标准 Agent？

- 场景：面试官质疑“只是工具调用，不算 Agent”
- 技术点：ReAct、tool-calling、状态管理、终止策略
- 回答提示：我们是“生产可控型 Agent”，不是开放式 AutoGPT；重点在可控执行与业务闭环。

### Q2：为什么用 LangGraph，不用纯手写循环？

- 场景：比较框架选型
- 技术点：checkpoint、resume、状态图可视化
- 回答提示：循环逻辑我们手写在 runtime，图只做稳定编排；两者结合兼顾灵活和治理。

### Q3：为什么不全部做成工具？

- 场景：工具化边界
- 技术点：能力分层、复杂度控制
- 回答提示：先把高价值且稳定的能力工具化，避免过早平台化导致维护成本飙升。

### Q4：工具参数错误怎么办？

- 场景：模型 hallucination 参数
- 技术点：schema 校验、一次自纠、错误回灌
- 回答提示：先返回 `INVALID_ARGUMENT` observation 给模型纠一次，再失败就终止。

### Q5：怎么避免无限调用工具？

- 场景：Agent 循环稳定性
- 技术点：max_steps、重复动作限制、空步限制
- 回答提示：三道刹车一起上，不把风险留给模型“自觉停止”。

### Q6：如何做可观测？

- 场景：线上问题定位
- 技术点：phase trace、结构化日志、会话维度
- 回答提示：能回放每一步决策链，知道是“模型错”还是“工具错”。

### Q7：RAG 为什么要三路融合？

- 场景：检索策略追问
- 技术点：语义召回与字面召回互补
- 回答提示：企业问题里“字面强约束”非常多，单向量会漏召回。

### Q8：为什么 chunk 级去重，不是文件级？

- 场景：证据质量
- 技术点：证据保留粒度
- 回答提示：文件级去重会丢关键片段，影响最终可解释性。

### Q9：流程图为什么要中断，不一步做完？

- 场景：人机分工
- 技术点：HITL interrupt/resume
- 回答提示：文件选择属于用户决策，模型不能替代确认。

### Q10：如果模型服务挂了会怎样？

- 场景：降级能力
- 技术点：degrade message、异常兜底
- 回答提示：不会让接口崩掉，给前端可消费错误或降级文本。

### Q11：权限怎么保证？

- 场景：安全追问
- 技术点：current_user 注入、工具上下文
- 回答提示：`user_id` 由后端注入，模型无法越权指定。

### Q12：这个系统最大的工程价值是什么？

- 场景：总结题
- 技术点：演进能力、稳定性、排障效率
- 回答提示：把“AI 智能”变成“可治理的工程能力”。

## 9. 可直接背诵的结尾 40 秒

这个项目我最核心的贡献，不是单点功能，而是把 AI 服务从“能跑”升级为“可生产”。  
具体来说，我搭建了统一的 Agent 执行闭环：模型决策、工具治理、RAG 证据、流式体验、中断恢复、全链路追踪。  
这样后续每加一个能力，不需要重构主链路，只要按工具协议接入就能稳定上线。  
这也是我做 AI 工程的思路：先治理，再扩展，让系统可持续演进。
