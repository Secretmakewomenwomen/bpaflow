# AI 服务 ReAct 面试高频问答

## 1. 这个项目你做的是什么

我做的是一套企业级 AI 服务底座，不是单纯接一个大模型接口。它能承接文件上传、知识入库、向量检索、多路召回、工具调用、多轮会话、SSE 流式输出，以及当前基于 ReAct 的 Agent Runtime 重构。

## 2. 这个 AI 服务解决什么问题

主要解决三类问题：

- 用户提问时，能从自己的资料中检索出可信证据
- AI 不只是回答，还能调用系统工具获取额外信息
- 整个 AI 服务具备持续扩展能力，而不是每加一个能力就重写主流程

## 3. 你说的“从 0 到 1”具体指什么

从 0 到 1 主要分三步：

1. 先把 AI 服务底座搭起来，包括文件上传、解析、切块、embedding、向量入库、检索、会话和流式返回
2. 再把规则路由升级成 model-driven tool-calling，让模型具备动作能力
3. 最后继续重构成标准化 ReAct Runtime，把状态、边界控制、终止条件和可观测性补齐

## 4. 为什么后端选 FastAPI

因为这个项目更像 AI orchestration 服务，而不是重事务业务系统。FastAPI 很适合做：

- AI 接口编排
- SSE 流式返回
- 快速定义 API
- 承接轻业务 + 强编排场景

## 5. 为什么数据库选 PostgreSQL

因为在 0 到 1 阶段，我更重视链路闭环和架构收敛。PostgreSQL 在这个项目里同时承担：

- 业务数据
- 文件元数据
- 会话消息
- 消息引用
- 向量检索
- BM25 检索
- Agent trace

这样可以把系统复杂度压到最低，开发、调试、部署和运维都更稳。

## 6. 为什么不用独立向量数据库

不是不能用，而是当前阶段没必要。一开始就拆独立向量库会引入：

- 多存储运维成本
- 数据同步问题
- 查询与调试复杂度
- 一致性问题

当前用 PostgreSQL + `pgvector` + `pg_search` 已经足够覆盖事务、向量和关键词检索需求。

## 7. 为什么检索不是只做向量检索

因为单路向量检索不够稳。它擅长语义相似，但对这些场景不够好：

- 文件名
- 编号
- 错误码
- 接口名
- 最近上传的 PDF 这类确定性需求

所以我做了三路召回：

- 向量召回
- BM25 召回
- 规则召回

最后做融合和 rerank。

## 8. 为什么一开始不是直接做 Agent

因为没有知识入库、检索、会话、引用和流式输出这些底座，Agent 只是空壳。真正落地 AI 服务，必须先把数据和链路打通。

## 9. 你们早期为什么用规则路由

因为规则路由适合快速落地，优点是：

- 可控
- 简单
- 风险低

但缺点是：

- 模型没有真正决策权
- 工具扩展成本高
- 后端会变成巨大的 if-else 中心

所以它更适合作为过渡阶段，而不是长期架构。

## 10. 什么是 tool-calling loop

可以简单理解成：

- 工具 schema 直接给模型
- 模型自己判断是否调用工具
- 模型决定调用哪个工具、调用几次
- 后端负责执行工具和处理错误

它解决的是“模型能不能做动作”的问题。

## 11. 为什么有了 tool-calling loop 还要做 ReAct

因为 tool-calling loop 只解决了“模型可以调工具”，但没有彻底解决：

- 状态怎么管理
- 工具调用前怎么做边界校验
- 什么时候停止
- 如何避免重复 action 和空转
- 如何记录每一步的 reason / action / observation
- runtime 如何从大 service 中解耦出来并具备清晰边界

ReAct 重构解决的是工程化运行能力，而不只是工具调用能力。

## 12. 你理解的 ReAct 是什么

我理解的 ReAct 不是一句 prompt 技巧，而是一种运行模式：

- Reason：先判断下一步该做什么
- Act：调用工具或执行动作
- Observe：接收外部结果
- Continue / Stop：继续推理或者终止

对于需要多步工具调用的企业级 AI 服务，这种模式比一次性回答更合适。

## 13. 为什么 ReAct 适合你们这个项目

因为我们的问题天然就是“边思考边行动”的问题。比如：

- 查知识库
- 查文件详情
- 查用户信息
- 触发流程图生成前的人机中断选择

这些都不是一次模型回答就能稳定完成的，更适合用 ReAct 逐步推进。

## 14. 为什么不一步到位做 planner-executor

因为当前业务阶段还没复杂到需要 planner-executor。这个结构更重，带来的复杂度包括：

- 计划状态管理
- 执行器协调
- 失败回滚
- 任务拆解与合并

当前阶段 ReAct 已经足够覆盖单助手、多步轻任务、工具调用和人机中断场景，是收益和复杂度更平衡的选择。

## 15. 为什么当前还保留 LangGraph

因为当前重构采用的是渐进式策略，不是大爆炸式重写。LangGraph 仍然有现实价值：

- 提供 graph shell
- 支持 streaming
- 支持 checkpoint

我做的是把 LangGraph 从“架构中心”降级成“一个 backend 和外层壳”，而不是一刀切掉。

## 16. 现在的核心架构是什么

当前核心架构可以概括为：

- `AgentFacade` 统一装配运行时组件
- Runtime 负责 ReAct 主循环
- Reasoning 负责模型决策
- Tools 负责协议和执行
- State 负责会话与 step 状态
- Guardrails 负责执行前校验
- Termination 负责停止条件
- Tracing 负责 step 级可观测

这是一个平台化 Runtime 架构，而不是单 service 编排。

## 17. `AgentFacade` 的作用是什么

它的作用是把运行时组件统一装配起来，并屏蔽上层对具体运行时细节的感知。这样：

- 上层接口不需要知道底层具体 runtime 细节
- 下层 runtime 可以继续演进
- 整体架构不会继续耦合回一个大 service

## 18. 为什么当前还保留 LangGraph runtime 外壳

当前主链路只有一套运行时实现，即 `LangGraphAgentRuntime`。

保留它的原因是：

- 现有 streaming 协议已经稳定接入
- checkpoint 能力仍然有现实价值
- 可以先把推理、工具、状态、安全边界这些核心抽象收口，再决定后续是否继续替换执行外壳

这样做的重点不是维持“双 runtime”，而是在不打断业务链路的前提下完成 ReAct 平台化收口。

## 19. Guardrails 解决什么问题

Guardrails 解决的是“模型建议不等于系统允许执行”的问题。它会在工具执行前校验：

- 用户上下文是否存在
- tool name 是否匹配
- 必填参数是否完整
- 是否需要确认

它的价值是给 AI 系统补上执行边界。

## 20. Termination 解决什么问题

Termination 解决的是“Agent 什么时候应该停”的问题。当前会考虑：

- 是否已经有 final response
- 是否进入 waiting input
- 是否达到最大步数
- 是否重复 action 太多
- 是否连续空转太多

这样可以避免模型无限 loop 或无效空转。

## 21. 为什么要做 tracing

因为 AI 系统的难点不只是结果对不对，而是错了以后你知不知道哪里错。step 级 tracing 可以记录：

- reason
- action
- observation
- termination

这样后续才能做排障、审计、调优和运营分析。

## 22. 为什么 trace 不和聊天消息存一起

因为它们职责不同：

- 聊天消息是用户可见数据
- trace 是运行态审计数据

聊天消息关心产品回放，trace 关心平台排障和过程审计，所以必须分层存储。

## 23. 这次 ReAct 重构最大的收益是什么

我一般会概括成四点：

1. 把 AI 服务从耦合的大 service 拆成清晰模块
2. 把工具调用从“功能点”升级成“平台协议”
3. 把停止条件和安全边界显式化
4. 把 AI 运行过程变成可观测、可审计、可调优

## 24. 这次重构最核心解决了什么问题

一句话说，就是把“模型能做事”升级成“系统能安全、稳定、可观测地让模型做事”。

## 25. 你觉得当前方案还有什么边界

当前边界主要有四个：

- 还保留 LangGraph 外壳，说明仍在渐进式重构阶段
- Memory 还是轻量短期记忆，没有真正长期记忆
- 还不是 planner-executor 架构
- 工具还是代码静态注册，不是动态工具平台

## 26. 如果继续演进，你下一步会做什么

我会分三层看：

- 近期：继续稳定 runtime、trace、tool 协议和更多业务工具
- 中期：补 planner-executor、长期记忆、更细的权限与确认机制
- 长期：演进到动态工具、MCP 接入、多 agent 协作平台

## 27. 如果面试官问你“这次设计最体现架构思维的地方是什么”

我会回答三点：

- 不是一开始就堆复杂架构，而是按业务阶段逐步演进
- 不是只看“能不能跑”，而是提前补齐控制、边界和观测
- 不是做一次性功能，而是把 AI 服务往平台型 Runtime 方向推进

## 28. 1 分钟收尾版

这个项目我最核心的工作，是把 AI 服务从一个功能型问答系统，逐步演进成一个平台型 Runtime。早期先用 PostgreSQL 单栈把知识入库、检索、会话和流式输出收敛住，然后从规则路由升级到 tool-calling，让模型拥有动作能力，最后再继续重构成 ReAct Runtime，把状态、安全、终止和 tracing 补齐。这样系统不只是能调用工具，而是能安全、稳定、可观测地调用工具，并且具备继续向 planner、多 agent 和长期记忆演进的基础。

## 29. 3 分钟收尾版

如果从架构演进看，这个项目我做了三层升级。第一层是 AI 底座建设，包括文件上传、解析、切块、embedding、向量检索、多路召回、会话落库和 SSE 流式输出。第二层是能力升级，从规则工具路由切到 model-driven tool-calling，让模型自己决定是否调用工具。第三层是平台升级，把 tool-calling loop 继续重构成 ReAct Runtime，拆出 Facade、Runtime、Reasoning、Tools、State、Guardrails、Termination 和 Tracing，使系统从“功能可用”进化到“工程可控、可观测、可演进”。这一步不是换个名字，而是把 AI 服务真正做成了具备长期演进能力的后端架构。
