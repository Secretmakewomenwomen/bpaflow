# AI 助手后端面试总结

## 适用场景

这份文档用于回答下面这类面试问题：

- 这个项目后端是怎么从 0 到 1 搭起来的
- 为什么后端技术栈这样选
- RAG、LangGraph、PostgreSQL 在项目里分别负责什么
- 你在这个 AI 助手里主要负责了哪些后端设计

---

## 一句话概括

我做的不是一个简单接大模型的 Demo，而是把 AI 助手真正落地需要的后端闭环搭起来了，包括知识入库、向量检索、Agent 编排、会话记忆、流式输出，以及基于 PostgreSQL 的统一持久化。

---

## 项目目标

这个 AI 助手面向工作台场景，核心目标有两类：

1. 根据用户问题检索知识库，返回命中的资料片段和相关文件
2. 基于检索结果继续做结构化生成，后续扩展到 XML 生成

在第一阶段，我优先落地的是第 1 类能力，也就是 RAG 检索闭环，但在架构设计上提前为 XML 生成预留了分支。

---

## 为什么这样选技术栈

后端技术栈我选的是：

- `FastAPI`：接口开发快，天然适合做 AI 服务编排层
- `PostgreSQL`：同时承担业务数据、会话数据、引用数据
- `pgvector`：直接在 PostgreSQL 里做向量检索，降低 0 到 1 阶段的系统复杂度
- `LangGraph`：做可扩展的 Agent 工作流编排，而不是把流程写死在一个大函数里
- `OSS`：存文件本体，数据库只存元信息和引用关系

我的核心考虑是先把系统收敛住。0 到 1 阶段最重要的是链路闭环和稳定落地，而不是一开始就把组件拆得很重，比如单独上一个向量数据库或者更复杂的调度系统。

---

## 后端从 0 到 1 的搭建过程

### 1. 先搭基础设施和启动链路

第一步我先把应用启动时需要的基础设施补齐，而不是先写 AI 逻辑。

启动时会做几件事：

- 初始化数据库连接
- 自动检查并创建业务表
- 自动检查并创建 `pgvector` 扩展
- 自动检查并创建向量表
- 自动初始化 LangGraph 的 checkpoint 表

这样做的好处是，新环境拉起服务时不需要手工补很多数据库动作，后端具备自举能力。

### 2. 先把知识库入库链路做通

RAG 不是从检索开始的，而是从“知识怎么入库”开始的。

我先把文件上传和知识入库链路做通，整体流程是：

1. 用户上传文件
2. 文件原始内容上传到 OSS
3. PostgreSQL 保存文件元数据
4. 后台异步触发向量化任务
5. 解析文本内容
6. 做清洗和分块
7. 调用 embedding 模型生成向量
8. 写入 pgvector 表

为什么这里要做成异步？

因为上传接口如果同步做解析和 embedding，用户等待时间会非常长，而且大文件容易阻塞接口。我的设计是上传请求先快速返回，后端在后台做向量化，前端通过文件状态感知当前是否可检索。

### 3. 解析层按文件类型拆开

在解析阶段，我没有用一个通用逻辑硬吃所有文件，而是按文件类型分流：

- `docx`：提取正文段落和表格行
- `pdf`：优先提取文本层，提取不到时走 OCR 兜底
- `png`：直接走 OCR，同时也支持图片向量化

这么做的原因是不同文件格式的结构差异非常大，如果统一粗暴处理，会直接影响后续切块质量和检索命中率。

### 4. 把检索能力做成独立服务

知识入库完成后，下一步才是查询检索。

用户提问后，检索流程是：

1. 先把 query 生成 embedding
2. 到 pgvector 文本表里查相似 chunk
3. 只检索当前用户自己的文件
4. 过滤低相似度结果
5. 聚合成两类结果：
   - 命中文本片段 `snippets`
   - 相关文件 `related_files`

这里我专门把“片段”和“文件”分开返回，而不是只返回一段 AI 回答。因为业务上用户不仅要知道答案，还要知道这个答案来自哪些文件，并且要能回看原始资料。

### 5. 用 LangGraph 做工作流编排

在 AI 编排层，我没有把逻辑写成一个 service 里的 if else，而是用 `LangGraph` 把整个流程拆成图。

当前图里的核心节点有：

- `load_history`
- `classify_intent`
- `retrieve_documents`
- `synthesize_answer`
- `generate_chat_answer`
- `build_response`
- `persist_message`

它的关键点在于先做意图识别，再决定走哪条链路：

- 普通聊天：直接走聊天生成
- 文档检索：走 RAG 检索和摘要生成
- XML 生成：当前先返回占位，后续直接补节点扩展

这样做的价值是后续可演进。第一阶段只做 RAG，但第二阶段要加 XML 生成时，不需要重写主流程，只需要补新的节点和分支。

### 6. 历史记忆拆成两层

我把“记忆”拆成了两层，这一点在面试里通常是亮点。

第一层是业务层记忆：

- `ai_conversation`
- `ai_message`
- `ai_message_reference`

这三张表负责保存可见的会话、消息、引用片段和相关文件，目的是给前端回放历史消息，也让用户能看到答案引用了哪些资料。

第二层是运行态记忆：

- 使用 `LangGraph PostgresSaver` 做 checkpoint

这层不是为了展示，而是为了保存图执行过程中的状态，让同一个 `conversation_id` 可以在服务重启后继续作为 LangGraph 的同一条线程执行。

我这样拆的原因是：

- 业务消息是用户可见数据
- checkpoint 是工作流运行态数据

它们的职责不同，所以不能混存成一张表。

### 7. 为前端提供同步接口和流式接口

在接口层，我同时提供了两种模式：

- 普通消息接口
- SSE 流式消息接口

流式接口的事件粒度是：

- `user_message`
- `assistant_start`
- `assistant_delta`
- `assistant_done`
- `error`

这样前端能做接近即时聊天的体验，而不是一直等待整段结果全部生成完再一次性返回。

## 我认为这个后端最有价值的几个设计点

### 1. PostgreSQL 同时承载业务数据和 AI 数据

我没有一开始就引入很多独立中间件，而是让 PostgreSQL 同时承担：

- 会话表
- 消息表
- 引用表
- 文件元数据
- 向量表
- checkpoint 相关表

这样 0 到 1 阶段架构足够简单，开发、调试、部署和运维成本都更低。

### 2. RAG 不是只返回答案，而是返回证据

我在响应结构里保留了片段和文件引用，这样 AI 输出具备可追溯性。这个设计比只给一个 answer 更符合业务场景，也更容易建立用户信任。

### 3. 用 LangGraph 保证后续扩展性

当前虽然主能力是检索，但系统从一开始就不是按“单轮问答接口”来设计的，而是按“可扩展 Agent 工作流”来设计的。后面要加 XML 生成、工具调用或者更复杂的分支，不需要推翻架构。

### 4. 记忆做了业务层和运行态分层

很多项目只做聊天记录存库，但没有把 Agent 的执行态持久化。我的实现里两层都考虑到了，所以多轮对话、断点恢复、后续复杂工作流更容易演进。

---

## 面试时可以直接说的 1 分钟版本

这个 AI 助手后端我是按完整闭环来搭的，不是只接了一个大模型接口。整体技术栈是 FastAPI、PostgreSQL、pgvector 和 LangGraph。先把文件上传到 OSS，再把元数据写 PostgreSQL，后台异步做解析、切块、embedding 和向量入库，这样知识库先建起来。用户提问后，先做意图识别，普通聊天走对话链路，查资料才走 RAG 检索链路。检索时我只查当前用户自己的向量数据，并且除了回答，还会返回命中的片段和相关文件。会话层面我做了两层记忆，一层是业务消息表，一层是 LangGraph 的 Postgres checkpoint，所以历史消息能回放，工作流状态也能续上。这样后面继续扩展 XML 生成或者更多 Agent 能力，架构不需要重做。

---

## 面试时可以直接说的 3 分钟版本

我在做这个 AI 助手后端时，思路是先把 AI 真正落地所需要的后端底座搭起来，而不是先堆一个问答接口。整个系统我拆成了五层：基础设施层、知识入库层、检索层、Agent 编排层、会话记忆层。

基础设施层我先做的是数据库和启动自举，服务启动时会自动检查业务表、向量表、pgvector 扩展和 LangGraph checkpoint 表，保证环境拉起足够稳定。知识入库层我先把文件上传和向量化流程打通，文件先上 OSS，数据库存元数据，然后后台异步做解析、清洗、分块和 embedding，最后写进 pgvector。这里按文件类型分别处理 docx、pdf 和 png，PDF 没有文本层时还会 OCR 兜底。

有了知识库以后，我再做检索层。用户提问后先生成 query embedding，再去 pgvector 里检索相似 chunk，并且严格按 user_id 做隔离。检索结果我不是只返回一段 answer，而是拆成 snippets 和 related_files 两类，因为业务上用户需要看到答案依据和原始文件入口。

在 AI 编排层我用了 LangGraph。图里现在有历史加载、意图识别、文档检索、摘要生成、普通聊天、响应构建和消息持久化几个节点。这样普通聊天不会误走 RAG，后面如果要接 XML 生成，也只需要新增分支节点，不用推翻已有结构。

记忆层我做了两层设计。第一层是业务消息表，保存会话、消息和引用关系，给前端做历史回放。第二层是 LangGraph 的 PostgresSaver，保存工作流运行态 checkpoint。这样不仅用户能看到历史消息，服务重启后同一个 conversation_id 也还能继续跑同一条 LangGraph 线程。

最后我还补了 SSE 流式接口，让前端能按 delta 渲染 AI 回复，聊天体验会更像真实助手而不是一次性结果面板。整体上我做的是一个可持续演进的 AI 后端底座，而不是一个一次性的问答 Demo。

## 面试官可能会追问的问题

### 为什么 0 到 1 阶段选 PostgreSQL + pgvector，而不是独立向量库？

因为 0 到 1 阶段最重要的是链路收敛和快速落地。PostgreSQL 既能存业务数据，也能存向量和会话数据，系统复杂度最低。等数据规模和检索复杂度进一步上升，再考虑拆分会更合理。

### 为什么要用 LangGraph？

因为这里不是单纯调用一次大模型，而是存在明确的状态流转和分支路由，比如普通聊天、RAG 检索、XML 生成。LangGraph 比一个大函数里的 if else 更适合长期演进和维护。

### 如果面试官问，为什么不用 `create_agent` 直接实现自动 loop？

可以这样回答：

我没有直接用 `create_agent` 去做默认自动 loop，核心原因不是它不能用，而是我们这个项目更偏企业级落地，要的不是“先把 agent 跑起来”，而是“把 agent 放进业务系统后仍然可控、可观测、可中断、可恢复”。

我最终自己显式实现 agent loop，主要基于下面几个考虑：

1. 我们需要的是“可控的 loop”，不是黑盒 loop  
   `create_agent` 很适合快速搭 PoC，但在正式项目里，我更关心每一轮工具调用能不能被观测、能不能打日志、能不能做 SSE 事件透出、能不能加缓存和熔断。自己实现 loop 后，我可以明确控制：
   - 每轮模型调用
   - 每次工具执行
   - `tool_start / tool_result` 这类前端事件
   - `max_turns` 这种保险丝
   - 参数错误后的自纠和终止策略

2. 我们项目里存在明显的 human-in-the-loop 中断点  
   比如“根据文件生成流程图”这条链路，不是模型自动跑完就结束，而是中间要暂停，等前端让用户选择文件，然后再恢复执行。这类“interrupt + resume”的业务状态，比纯自动 agent 更复杂。自己实现 loop，更容易把这类等待输入、恢复执行的状态放进同一条后端主链路里。

3. 我们的任务不是全都适合自由 agent  
   项目里同时存在两类任务：
   - 开放式检索和问答，适合 agent loop
   - 根据文件生成流程图 JSON，这类目标清晰、步骤稳定的任务，更适合 workflow

   如果直接依赖 `create_agent` 的默认自动 loop，很容易把所有任务都塞进自由决策模式里，后面会遇到收敛不稳、局部循环、行为不可控的问题。自己实现 loop，可以更方便做“按场景分流”。

4. 我们前端不是只等一个最终答案，而是依赖完整状态流  
   在这个项目里，前端除了展示回答，还要感知：
   - 工具是否开始执行
   - 工具是否返回成功
   - 是否进入 waiting_input
   - 是否返回动作按钮，比如“导入”“重新选择文件”

   这意味着后端输出的不只是一个模型结果，而是一套和前端交互协议绑定的状态流。自己实现 loop，能让前后端协议保持稳定，而不是依赖通用 agent 的默认输出格式。

5. 我们需要对局部循环和过度调用做工程治理  
   实际落地里，我遇到过多主题检索时 agent 围绕某一个文件反复检索的问题。所以我更希望 loop 的控制权掌握在自己手里，方便继续加这些治理能力：
   - `max_turns`
   - 同参数工具缓存
   - 相似 query 去重
   - 多主题先汇总再深挖的约束
   - 无新增结果时的熔断

面试时可以再补一句总结：

`create_agent` 更适合快速验证通用 agent 能力，而我们这个项目的重点是把 agent 嵌进企业工作台的业务流程里，所以我更看重工程可控性，而不是默认自动化能力。不是 `create_agent` 不行，而是当前场景下，自己实现显式 loop 更适合长期维护和业务扩展。

### 为什么聊天记录和 checkpoint 不放一套？

因为它们的职责不同。聊天记录是给用户看的业务数据，checkpoint 是给工作流续跑用的运行态数据。混在一起会让模型层和业务层耦合得很重。

### 这个项目里你觉得最关键的后端难点是什么？

我觉得不是“把模型调通”，而是把知识入库、检索、引用回传、记忆持久化和工作流编排真正串成一个闭环，并且保证后面还能继续扩展。

## 最后一句总结

这个项目的后端 0 到 1，不是“先有模型，再去补业务”，而是先把业务系统里的 AI 基础设施搭起来，再把模型能力挂进去。这样做出来的是可落地、可扩展、可维护的 AI 助手后端。
