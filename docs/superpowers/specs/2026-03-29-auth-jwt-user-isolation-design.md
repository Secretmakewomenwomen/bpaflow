# 登录注册与 JWT 用户隔离设计

## 背景

当前项目前端是单页画布工作台，后端提供上传与工作相关接口，但没有用户体系、登录态和按用户隔离的数据边界。新增认证能力后，需要保证：

- 用户可使用用户名和密码完成注册与登录
- 每个用户拥有独立 `user_id`，格式为 UUID
- 只有登录成功后才允许进入画布页面
- 现有业务数据按 `user_id` 隔离，不允许跨用户读取或删除

## 本次范围

本次迭代包含以下内容：

- 前端新增登录页和受保护路由
- 前端集成 `ant-design-vue`
- 后端新增用户表、密码校验、JWT 签发与鉴权
- 现有上传和工作数据模型增加 `user_id`
- 现有上传和工作接口改为仅返回当前登录用户的数据
- 新增认证相关测试和跨用户隔离测试

本次迭代不包含以下内容：

- 找回密码
- 刷新 token
- 角色权限体系
- 第三方登录
- 多租户后台管理

## 目标

### 用户目标

- 未登录用户访问应用时先进入登录/注册页
- 注册成功后自动登录并进入画布
- 登录成功后进入画布
- 页面刷新后可恢复登录态
- 退出登录后回到登录页

### 系统目标

- 对外只暴露 UUID 格式的 `user_id`
- 所有业务接口基于 JWT 中的当前用户身份执行
- 业务接口不接受前端显式传入的 `user_id`
- 同一接口在不同用户上下文下只能访问各自的数据

## 方案对比

### 方案 A：JWT 解析当前用户，后端统一按 `user_id` 过滤

这是本次采用的方案。

优点：

- 安全边界清晰，前端无法通过篡改参数访问其他用户数据
- 业务接口调用方式统一，前端无需传 `user_id`
- 后端扩展新接口时更容易复用鉴权依赖

缺点：

- 需要补齐认证基础设施
- 需要调整现有表结构与测试

### 方案 B：前端显式传 `user_id`

不采用。

原因：

- 前端容易漏传
- 需要每个接口重复校验
- 风险高，容易出现越权读取

## 总体设计

### 前端结构

前端从当前单页结构调整为路由化结构：

- `/login`：登录/注册页，公开访问
- `/canvas`：画布页，受保护

新增模块：

- `src/router`：路由定义与守卫
- `src/pages/AuthPage.vue`：登录注册页面
- `src/pages/CanvasPage.vue`：承载现有画布工作台
- `src/lib/auth.ts`：token 保存、读取、清理、当前用户恢复
- `src/lib/http.ts`：统一请求封装，自动附带 `Authorization`

现有 `src/App.vue` 降级为路由承载入口，原有画布主体迁移到 `CanvasPage.vue`。

### 后端结构

后端在 FastAPI 现有结构上新增：

- `app/models/user.py`
- `app/schemas/auth.py`
- `app/api/routes/auth.py`
- `app/core/security.py`
- `app/dependencies/auth.py`

现有业务模型和路由改造：

- `upload` 相关模型和接口增加 `user_id`
- `work` 相关模型和接口增加 `user_id`
- 所有业务写接口自动写入当前用户 `user_id`
- 所有业务读和删接口基于当前用户过滤

## 数据模型设计

### users 表

建议字段：

- `id`: bigint/int，自增主键，仅内部使用
- `user_id`: varchar(36)，UUID，唯一索引，对外暴露
- `username`: varchar，唯一索引
- `password_hash`: varchar，不存明文密码
- `created_at`: datetime

约束：

- `user_id` 唯一
- `username` 唯一，注册时禁止重复

### 业务表改造

现有需要隔离的业务表均增加 `user_id` 字段。

当前已确认范围：

- 上传记录表
- 工作记录表

约束建议：

- `user_id` 非空
- 对高频查询表建立 `(user_id, created_at)` 或 `(user_id, id)` 组合索引

迁移策略：

- 对空库直接建新字段
- 若已有历史数据，可在本地开发阶段用默认脚本回填到测试用户；正式环境再补迁移策略

## 认证设计

### 注册

接口：`POST /api/auth/register`

请求字段：

- `username`
- `password`

处理流程：

1. 校验用户名不能为空，长度在约束范围内
2. 校验密码不能为空，长度满足最小要求
3. 查询用户名是否已存在
4. 生成 UUID 作为 `user_id`
5. 对密码做 hash
6. 写入 `users` 表
7. 签发 JWT
8. 返回 token 和当前用户信息

### 登录

接口：`POST /api/auth/login`

请求字段：

- `username`
- `password`

处理流程：

1. 按用户名查询用户
2. 校验密码 hash
3. 签发 JWT
4. 返回 token 和当前用户信息

### 当前用户

接口：`GET /api/auth/me`

处理流程：

1. 从请求头解析 Bearer Token
2. 校验 JWT 合法性和过期时间
3. 返回当前用户公开信息

### 退出登录

本次采用前端本地退出：

- 前端删除本地 token
- 清空当前用户状态
- 跳转到 `/login`

后端不做 token 黑名单。

## JWT 设计

### Payload

最小字段：

- `sub`: `user_id`
- `username`
- `exp`

### 约束

- 前端仅保存 access token
- 后端所有受保护接口统一通过依赖解析 token
- 业务代码不直接解析请求头，避免重复逻辑

### 密码处理

密码必须做 hash 存储，不允许明文入库。优先使用成熟密码哈希方案。

## 前端交互设计

### 登录注册页

使用 `ant-design-vue` 组件：

- `Tabs`
- `Form`
- `Input`
- `InputPassword`
- `Button`
- `Alert`
- `Card`

页面包含两个 Tab：

- 登录
- 注册

登录表单字段：

- 用户名
- 密码

注册表单字段：

- 用户名
- 密码
- 确认密码

交互规则：

- 表单校验失败时在前端直接提示
- 接口失败时显示错误信息
- 注册成功后直接进入画布
- 登录成功后直接进入画布

### 画布页

画布页继续复用现有工作台结构，仅增加登录态相关展示：

- 顶部显示当前用户名
- 顶部提供退出登录按钮

### 路由守卫

规则如下：

- 未登录访问 `/canvas`，重定向到 `/login`
- 已登录访问 `/login`，重定向到 `/canvas`
- 启动时若本地存在 token，先恢复用户态再决定路由去向

## 后端接口设计

### 新增接口

- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/auth/me`

### 改造接口

以下现有接口改为受保护接口，并自动按当前用户过滤：

- 上传创建
- 上传列表
- 上传删除
- 工作创建
- 工作列表
- 工作删除

### 鉴权依赖

提供统一依赖 `get_current_user`：

- 负责解析 JWT
- 负责加载当前用户
- 供认证接口以外的业务路由复用

## 错误处理

认证相关：

- 用户名已存在：返回 400
- 用户名或密码错误：返回 401
- token 缺失或无效：返回 401

数据隔离相关：

- 删除不存在或不属于当前用户的数据：返回 404
- 列表查询只返回当前用户可见数据，不暴露其他用户记录

## 测试设计

后端新增或调整以下测试：

- 注册成功
- 重复用户名注册失败
- 登录成功
- 错误密码登录失败
- `/api/auth/me` 在合法 token 下返回当前用户
- 不带 token 访问受保护接口失败
- 用户 A 创建上传记录，用户 B 无法在列表中看到
- 用户 A 创建工作记录，用户 B 无法在列表中看到
- 用户 B 删除用户 A 的记录时返回 404

前端至少验证：

- 未登录无法进入画布
- 登录成功跳转到画布
- 注册成功跳转到画布
- 退出登录后跳转回登录页

## 实施顺序

1. 引入前端依赖：`vue-router`、`ant-design-vue`
2. 新增前端路由与认证模块
3. 拆分现有 `App.vue` 为登录页和画布页
4. 新增后端用户模型、认证 schema、认证路由和安全工具
5. 改造上传与工作模型，补 `user_id`
6. 改造上传与工作接口为受保护接口
7. 更新前端请求封装，统一携带 token
8. 补认证测试与跨用户隔离测试

## 风险与注意事项

- 当前项目若已有本地历史数据，增加 `user_id` 后需要迁移或清理旧数据
- 如果后续要扩展更多业务对象，需要坚持“从 JWT 取当前用户”的边界，避免重新引入前端传 `user_id`
- 若后续需要长期登录体验，可再补 refresh token，但不纳入本次实现

## 验收标准

- 用户可完成注册和登录
- 未登录时无法进入画布页
- 登录后刷新页面不丢失登录态
- 顶部可展示当前用户名并支持退出
- 上传与工作接口只返回当前用户数据
- 不同用户之间不能读取或删除彼此的数据
