# 企业级 Agent 差距清单

## 目的

本文用于评估当前 backend AI 服务是否已经达到企业级 Agent 模式，并整理现阶段与目标状态之间的关键差距，作为后续建设和排期依据。

## 当前结论

当前系统已经具备可运行的 ReAct Agent 主循环，当前主链路运行在 LangGraph-backed ReAct runtime 上。  
但从工程成熟度来看，现阶段更准确的定位应为：

- 可工作的 ReAct Agent Runtime v1
- 企业级 Agent 平台骨架
- 尚未达到完整企业级 Agent 平台标准

问题不在于“有没有 agent loop”，而在于平台治理、稳定性、恢复能力、观测能力和运维能力还没有完全闭环。

## 已具备能力

- 已具备 ReAct 主循环
- 已具备工具调用协议
- 已具备基础 guardrails
- 已具备 termination 控制
- 已具备 reasoning/tool trace 基础能力
- 已具备流式输出能力
- 已具备基础的人机中断处理能力

这些能力说明当前系统已经不是普通 chat service，而是一个具备 agent runtime 结构的后端 AI 服务。

## 差距清单

### 1. Agent API 尚未真正对外完成

当前 `/agent` 路由仍未接入实际 service，依然返回未配置状态。  
这意味着平台化接口仍未真正完成，外部系统无法以统一 agent API 的方式稳定接入。

影响：

- 无法对外宣称 agent 平台接口完成
- 无法形成统一线程、恢复、事件流协议
- 后续接入方仍需依赖兼容链路

### 2. 主链路仍依赖兼容层

当前实际对话主入口仍通过 `LangGraphAssistantService` 包装后再委托到 `AgentFacade`。  
这说明平台边界虽然已经出现，但主入口尚未完全收口到统一 agent 平台接口。

影响：

- 架构职责仍有历史包袱
- 平台接口与兼容接口并存，后续维护成本偏高
- 新旧链路混用时更容易出现行为不一致

### 3. 会话恢复与断点续跑能力未形成完整闭环

企业级 agent 不仅要能跑，还要能恢复。  
当前虽然已有部分中断与恢复设计，但还缺少完整的 step 级恢复协议验证，包括：

- 工具执行中断后的恢复
- pending action 的恢复
- resume 的幂等保障
- checkpoint 与消息状态的一致性校验

影响：

- 服务中断后可能无法稳定续跑
- 用户重复触发恢复操作时可能产生重复执行
- 线上问题排查复杂度高

### 4. 缺少生产级观测指标体系

当前已有 trace 基础，但企业级要求的不只是日志和 trace，还需要统一指标体系。

建议补充：

- agent 成功率
- final answer 达成率
- 平均 step 数
- tool 调用成功率
- tool 错误分布
- 重试率
- 超时率
- 中断恢复成功率
- token 消耗统计
- 端到端时延分布

影响：

- 无法量化 agent 质量
- 无法做版本对比和效果回归
- 出现线上问题时告警和定位能力不足

### 5. 超时、熔断、降级策略不足

企业级 agent 必须具备明确的失败控制策略。  
当前工具执行和模型推理具备基础错误处理，但还不足以覆盖生产环境中的复杂故障。

建议补充：

- 模型调用超时
- 工具调用超时
- 服务级熔断
- 限流
- 降级回答策略
- fallback runtime 或 fallback answer

影响：

- 外部依赖抖动时容易卡死
- 错误容易沿主链路放大
- 高峰期稳定性不可控

### 6. 权限与租户隔离仍需加强

当前已有 user context 和参数校验，但企业级场景通常需要更严格的权限模型：

- 工具级权限
- 资源级权限
- 用户与数据归属校验
- 审计字段完整落库
- 默认拒绝原则

影响：

- 工具调用可能存在越权风险
- 后续扩展更多工具时风险会继续放大
- 审计与合规能力不足

### 7. 幂等与并发控制不足

企业级 agent 必须处理重复提交、SSE 重连、多副本消费和用户重复点击等情况。

建议补充：

- 同一会话同一步骤的防重入
- 工具执行幂等 key
- resume 请求防重复执行
- assistant message 持久化防重复落库
- 多实例部署下的状态竞争控制

影响：

- 可能重复执行工具
- 可能重复写消息或 trace
- 并发场景下行为难以预测

### 8. 测试体系仍不足以支撑企业级上线

当前更像是功能实现阶段的测试覆盖，距离企业级上线测试要求还有差距。

建议补充的测试类型：

- 多轮 ReAct 回归测试
- 工具失败与重试测试
- checkpoint 恢复测试
- 权限绕过测试
- 并发与重入测试
- SSE 流式协议测试
- 长对话上下文退化测试
- 压测与故障注入测试

影响：

- 改动后容易出现隐性回归
- 无法对稳定性做有证据的承诺

### 9. Prompt 与 Tool 缺少版本治理能力

企业级 agent 的 prompt、tool schema、routing policy 都应具备版本化和灰度能力。

建议补充：

- prompt version
- tool schema version
- runtime 开关
- 灰度发布能力
- 快速回滚能力

影响：

- 线上问题难以定位到具体版本
- prompt 调整后缺少稳定的对比依据
- 风险控制成本高

### 10. 运维与审计能力仍需补齐

企业级平台不仅要能运行，还要能被运维、被审计、被复盘。

建议补充：

- 可检索的 step trace 查询能力
- 工具输入输出摘要检索
- 错误归因面板
- 人工介入记录
- 敏感信息脱敏
- 审计导出能力

影响：

- 出问题后恢复慢
- 很难支持长期运营
- 难以满足审计与合规要求

## 成熟度判断

如果按工程成熟度分级，可以给出如下判断：

- 架构形态：中上
- 可运行性：中上
- 企业级成熟度：中

因此，当前更适合对外表述为：

- 已完成 ReAct Agent Runtime v1
- 已形成企业级 Agent 平台骨架
- 尚未完成企业级 Agent 平台闭环

## 优先级建议

### P0

- 打通 `/agent` 正式入口
- 建立恢复、重试、幂等闭环
- 补充生产指标与告警
- 补齐高优先级回归测试

### P1

- 收口统一平台入口，减少兼容层职责
- 建立超时、熔断、降级策略
- 强化权限模型与租户隔离
- 增加并发与重复执行控制

### P2

- 建立 prompt/tool/routing version 治理
- 建设运维面板和审计检索能力
- 完善长期运营所需的数据分析能力

## 总结

当前 backend AI 服务已经进入 Agent 形态，不再只是传统对话服务。  
但距离“企业级 Agent 平台”还有明显差距，核心短板集中在：

- 平台化接口未完全收口
- 生产稳定性策略不足
- 恢复与幂等闭环不足
- 观测与运维能力不足
- 测试与治理体系不足

因此，现阶段最准确的判断应为：

当前已达到可运行的 ReAct Agent Runtime 阶段，但尚未达到完整企业级 Agent 平台阶段。
